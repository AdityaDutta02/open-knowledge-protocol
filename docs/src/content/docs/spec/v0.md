# Open Knowledge Protocol Specification — v0

**Status:** Draft  
**Author:** Aditya Dutta  
**Version:** 0.0.1-preview  
**Date:** 2026-04-06

---

## Abstract

The Open Knowledge Protocol (OKP) is an open specification for publishing web content in a form that AI agents can understand, traverse, and reason about. Where existing web standards treat documents as isolated text objects, OKP treats them as nodes in a typed directed graph — each node carrying semantic identity, relational context, temporal validity, and confidence metadata. OKP defines a canonical data structure called ConceptDNA, which embeds machine-readable knowledge metadata directly in every article. It defines a graph container called ConceptGraph for multi-node traversal. It specifies three transport profiles — MCP, REST, and llms.txt — for exposing that graph to different classes of AI consumers. And it defines a five-dimensional compliance scoring system so publishers can measure and signal their protocol adherence level. OKP is transport-agnostic and CMS-agnostic: any publisher who can embed a JSON object in their content pipeline can implement it. The v0 specification carries no stability guarantees; the design is being validated in production on The Context Window before v1 is locked.

## 1. Problem Statement

The World Wide Web was designed to serve human readers navigating with browsers. It was later extended to serve search engine crawlers through structured data markup (schema.org, JSON-LD), sitemaps, and RSS feeds. But neither of those audiences is the primary consumer of web content today for many purposes. AI language models — both during training and at inference time — ingest enormous volumes of web text. And increasingly, AI agents are querying live web content to answer questions, perform research, and synthesize knowledge across sources. The web was not designed for them, and the mismatch is structural.

The first deficiency is the absence of semantic identity. An article about "transformer architecture" sits as a blob of HTML with a URL. There is no machine-readable identifier that places this article into the broader graph of AI concepts. There is no way for an agent to know whether this article covers the same "transformer" as another article on a different domain, or whether it is about the architecture pattern in software engineering. Without a stable conceptId, knowledge cannot be linked across publishers, deduplicated, or versioned. Each article is an island with no address in concept-space.

The second deficiency is the absence of relational context. Human readers follow hyperlinks between articles, but those links are untyped: the link from "attention mechanism" to "transformer" could mean prerequisite, example, history, contradiction, or tangent — the reader must infer from prose. AI agents need typed relationships to build reasoning chains. Without explicit prerequisite/enables/relatedTo edges, an agent cannot determine the correct reading order, cannot identify knowledge gaps, and cannot distinguish foundational concepts from peripheral ones.

The third deficiency is the absence of temporal validity. AI agents cannot determine from standard web markup whether a given article is current, outdated, superseded, or speculative. A page about "GPT-3 capabilities" published in 2020 looks structurally identical to one published in 2025. There is no machine-readable expiry date, no review schedule, no explicit signal that newer content supersedes it. Agents that synthesize information across multiple documents have no principled basis for temporal ranking, which leads to confident hallucinations blending stale and current claims.

The fourth deficiency is the absence of confidence metadata. Web content exists on a spectrum from peer-reviewed fact to informed speculation to outright fabrication, but that spectrum is not declared. Standard metadata vocabularies (Dublin Core, schema.org) offer author, date, and license fields, but not epistemic status. An AI agent ingesting claims from a "deep analysis" and from "early speculation" needs to know which is which before weighting them in a synthesis. Without declared confidence levels and article type classifications, the agent has no basis for calibrated reasoning.

The fifth deficiency is the absence of graph connectivity. Even when sites use structured markup, each site's knowledge graph is a closed silo. There is no cross-publisher edge format, no protocol for traversing from a node on Site A to a related node on Site B, and no standard for exposing the graph topology via a queryable API. AI agents cannot programmatically explore concept neighborhoods, trace prerequisite chains, or identify contradictions across publishers. The knowledge embedded in the web exists as a vast, untraversable latent structure.

OKP addresses all five deficiencies with a single coherent design: a typed, temporally-aware, confidence-annotated knowledge graph with standardized transport interfaces. It does not replace existing standards — it builds on them. The goal is to make the web's latent knowledge structure explicit, machine-readable, and traversable by the AI agents that increasingly need to reason across it.

## 2. Core Abstractions

OKP defines three core data structures: ConceptDNA, KnowledgeNode, and ConceptGraph. These are the foundational abstractions on which all transport profiles and compliance scoring are built. Implementations must faithfully represent all required fields; optional fields should be included wherever the content supports them, as they significantly improve the utility of the graph for reasoning agents.

### 2.1 ConceptDNA

ConceptDNA is the semantic identity metadata for a single knowledge article. It is the core contribution of OKP: a structured envelope that every published article carries, declaring its identity, relationships, temporal validity, and epistemic status in machine-readable form. ConceptDNA is not a summary of the article — it is the article's position in the knowledge graph.

The following table defines all ConceptDNA fields. Fields marked Required must be present for any OKP-compliant payload.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `conceptId` | `string` | Yes | Unique kebab-case identifier within publisher's graph. Must be stable. |
| `title` | `string` | Yes | Human-readable article title. |
| `summary` | `string` (max 500 chars) | Yes | One-to-two sentence summary for AI agent consumption. |
| `keyTerms` | `string[]` (min 1) | Yes | Primary concepts this article defines or explains in depth. |
| `prerequisites` | `string[]` | Yes | conceptIds the reader must understand before this article. |
| `enables` | `string[]` | Yes | conceptIds whose understanding this article unlocks. |
| `relatedTo` | `string[]` | Yes | Laterally related conceptIds. |
| `category` | `string` | Yes | Topic cluster this article belongs to. |
| `confidence` | `'current' \| 'outdated' \| 'disputed' \| 'speculative'` | Yes | Reliability of the content at time of publication. |
| `temporalValidity` | `{ publishedAt: string, reviewBy?: string, expiresAt?: string }` | Yes | ISO 8601 datetimes for publication, scheduled review, and definitive expiry. |
| `articleType` | `'deep-dive' \| 'primer' \| 'prediction' \| 'analysis' \| 'news'` | Yes | Content classification. |
| `predictions` | `Prediction[]` | No | Forward-looking claims with confidence scores and resolution status. |
| `contradicts` | `string[]` | No | conceptIds whose claims conflict with this article's claims. |
| `supersedes` | `string[]` | No | conceptIds that this article replaces or updates. |
| `citations` | `Citation[]` | No | Sources supporting the article's claims. |

### 2.2 KnowledgeNode

A KnowledgeNode is a ConceptDNA instance bound to a published URL.

```typescript
interface KnowledgeNode {
  conceptId: string;
  title: string;
  url: string;         // canonical HTTPS URL
  conceptDNA: ConceptDNA;
}
```

### 2.3 ConceptGraph

A ConceptGraph is a multi-node graph returned in response to traversal queries. Formal definition: **G = (V, E, τ)** where V is the set of KnowledgeNodes, E is the set of directed edges, and τ assigns an EdgeType to each edge.

EdgeType values: `prerequisite`, `enables`, `relatedTo`, `contradicts`, `supersedes`.

## 3. Compliance Tiers

| Tier | Required composite score | Typical implementation |
|------|--------------------------|------------------------|
| Bronze | ≥ 50 | All required ConceptDNA fields present, at least one edge type populated |
| Silver | ≥ 70 | Required fields present, temporal validity complete, multiple edge types populated |
| Gold | ≥ 90 | All required and most optional fields present, rich graph connectivity, regular `confidence` updates |

Five scoring dimensions: `semantic-identity`, `relational-context`, `temporal-validity`, `confidence-metadata`, `graph-connectivity`. Composite score is the arithmetic mean of all five.

## 4. Transport Profiles

### 4.1 MCP Transport

Five standard tools at `/api/mcp`:

| Tool | Signature |
|------|-----------|
| `search_articles` | `(query: string, limit?: number)` |
| `get_article` | `(conceptId: string)` |
| `get_related` | `(conceptId: string, limit?: number)` |
| `get_prerequisites` | `(conceptId: string)` |
| `get_graph` | `(conceptId: string, depth?: number)` |

### 4.2 REST Transport

```
GET /api/okp/v0/article?conceptId=<id>
GET /api/okp/v0/graph?conceptId=<id>&depth=<n>
```

### 4.3 llms.txt Transport

`/llms.txt` — compact index with one-line summaries.  
`/llms-full.txt` — full content with embedded ConceptDNA.

## 5. Versioning Policy

Current version: `0.0.1-preview`. No stability guarantees in v0. v1 target: 2026 Q4, after production validation on at least two independent deployments.

Adding optional fields is backwards-compatible. Adding or removing required fields, renaming fields, or changing EdgeType enum values are breaking changes requiring a major version bump.

---

## 6. Relationship to Existing Standards

| Standard | What OKP adds |
|----------|---------------|
| RSS 2.0 | Knowledge structure: typed relationships, graph traversal, conceptIds |
| JSON-LD / schema.org | Relational graph topology, confidence levels, temporal validity |
| llms.txt | Semantic content typing, prerequisite/enables edges, graph queryability |
| MCP | Knowledge payload semantics — what tools return, not how they are invoked |
| Dublin Core | Relational context, epistemic confidence, AI-native fields |
