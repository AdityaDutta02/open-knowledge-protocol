# Open Knowledge Protocol Specification — v0

**Status:** Draft  
**Author:** Aditya Dutta  
**Version:** 0.0.1-preview  
**Date:** 2026-04-06

---

## Abstract

The Open Knowledge Protocol (OKP) is an open specification for publishing web content in a form that AI agents can understand, traverse, and reason about. Where existing web standards treat documents as isolated text objects, OKP treats them as nodes in a typed directed graph — each node carrying semantic identity, relational context, temporal validity, and confidence metadata. OKP defines a canonical data structure called ConceptDNA, which embeds machine-readable knowledge metadata directly in every article. It defines a graph container called ConceptGraph for multi-node traversal. It specifies three transport profiles — MCP, REST, and llms.txt — for exposing that graph to different classes of AI consumers. And it defines a five-dimensional compliance scoring system so publishers can measure and signal their protocol adherence level. OKP is transport-agnostic and CMS-agnostic: any publisher who can embed a JSON object in their content pipeline can implement it. The v0 specification carries no stability guarantees; the design is being validated in production on The Context Window before v1 is locked.

## 1. Problem Statement

The World Wide Web was designed to serve human readers navigating with browsers. It was later extended to serve search engine crawlers through structured data markup (schema.org, JSON-LD), sitemaps, and RSS feeds. But neither of those audiences is the primary consumer of web content today for many purposes. AI language models — both during training and at inference time — ingest enormous volumes of web text. And increasingly, AI agents are querying live web content to answer questions, perform research, and synthesize knowledge across sources. The web was not designed for them, and the mismatch is structural.

The first deficiency is the absence of semantic identity. An article about "transformer architecture" sits as a blob of HTML with a URL. There is no machine-readable identifier that places this article into the broader graph of AI concepts. There is no way for an agent to know whether this article covers the same "transformer" as another article on a different domain, or whether it is about the architecture pattern in software engineering. Without a stable conceptId, knowledge cannot be linked across publishers, deduplicated, or versioned. Each article is an island with no address in concept-space.

The second deficiency is the absence of relational context. Human readers follow hyperlinks between articles, but those links are untyped: the link from "attention mechanism" to "transformer" could mean prerequisite, example, history, contradiction, or tangent — the reader must infer from prose. AI agents need typed relationships to build reasoning chains. Without explicit prerequisite/enables/relatedTo edges, an agent cannot determine the correct reading order, cannot identify knowledge gaps, and cannot distinguish foundational concepts from peripheral ones.

The third deficiency is the absence of temporal validity. AI agents cannot determine from standard web markup whether a given article is current, outdated, superseded, or speculative. A page about "GPT-3 capabilities" published in 2020 looks structurally identical to one published in 2025. There is no machine-readable expiry date, no review schedule, no explicit signal that newer content supersedes it. Agents that synthesize information across multiple documents have no principled basis for temporal ranking, which leads to confident hallucinations blending stale and current claims.

The fourth deficiency is the absence of confidence metadata. Web content exists on a spectrum from peer-reviewed fact to informed speculation to outright fabrication, but that spectrum is not declared. Standard metadata vocabularies (Dublin Core, schema.org) offer author, date, and license fields, but not epistemic status. An AI agent ingesting claims from a "deep analysis" and from "early speculation" needs to know which is which before weighting them in a synthesis. Without declared confidence levels and article type classifications, the agent has no basis for calibrated reasoning.

The fifth deficiency is the absence of graph connectivity. Even when sites use structured markup, each site's knowledge graph is a closed silo. There is no cross-publisher edge format, no protocol for traversing from a node on Site A to a related node on Site B, and no standard for exposing the graph topology via a queryable API. AI agents cannot programmatically explore concept neighborhoods, trace prerequisite chains, or identify contradictions across publishers. The knowledge embedded in the web exists as a vast, untraversable latent structure.

OKP addresses all five deficiencies with a single coherent design: a typed, temporally-aware, confidence-annotated knowledge graph with standardized transport interfaces. It does not replace existing standards — it builds on them. The goal is to make the web's latent knowledge structure explicit, machine-readable, and traversable by the AI agents that increasingly need to reason across it.

## 2. Core Abstractions

OKP defines three core data structures: ConceptDNA, KnowledgeNode, and ConceptGraph. These are the foundational abstractions on which all transport profiles and compliance scoring are built. Implementations must faithfully represent all required fields; optional fields should be included wherever the content supports them, as they significantly improve the utility of the graph for reasoning agents.

### 2.1 ConceptDNA

ConceptDNA is the semantic identity metadata for a single knowledge article. It is the core contribution of OKP: a structured envelope that every published article carries, declaring its identity, relationships, temporal validity, and epistemic status in machine-readable form. ConceptDNA is not a summary of the article — it is the article's position in the knowledge graph. An AI agent that reads only the ConceptDNA of an article can make informed decisions about whether to read the full text, whether it already has sufficient context from prerequisites, and how to weight the article's claims against others.

The canonical TypeScript interface for ConceptDNA is the normative definition. All transport profiles serialize and deserialize against this interface. Where field names differ from existing metadata vocabularies (e.g., Dublin Core's `dc:identifier` vs. OKP's `conceptId`), OKP's names take precedence within OKP contexts; implementers are free to provide both.

The following table defines all ConceptDNA fields. Fields marked Required must be present for any OKP-compliant payload. Optional fields may be omitted if not applicable, but their absence reduces the compliance score in the relevant dimension.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `conceptId` | `string` | Yes | Unique kebab-case identifier within publisher's graph (e.g., `transformer-architecture`). Must be stable — changing a conceptId is a breaking change for all downstream edges. |
| `title` | `string` | Yes | Human-readable article title. Should match the canonical `<h1>` of the article. |
| `summary` | `string` (max 500 chars) | Yes | One-to-two sentence summary optimized for AI agent consumption. Should be information-dense and declarative rather than promotional. |
| `keyTerms` | `string[]` (min 1) | Yes | Primary concepts this article defines or explains in depth. Not a tag cloud — these are the concepts an agent can cite this article as a source for. |
| `prerequisites` | `string[]` | Yes | conceptIds the reader must understand before this article. Empty array `[]` is valid for foundational articles. |
| `enables` | `string[]` | Yes | conceptIds whose understanding this article unlocks. Together with `prerequisites`, this forms the directed learning graph. |
| `relatedTo` | `string[]` | Yes | Laterally related conceptIds — neither prerequisite nor dependent. Useful for concept neighborhood traversal. |
| `category` | `string` | Yes | Topic cluster this article belongs to (e.g., `machine-learning`, `deployment`, `security`). Used for graph partitioning and filtered search. |
| `confidence` | `'current' \| 'outdated' \| 'disputed' \| 'speculative'` | Yes | Reliability of the content at time of publication. Publishers must update this field when content ages. |
| `temporalValidity` | `{ publishedAt: string, reviewBy?: string, expiresAt?: string }` | Yes | ISO 8601 datetimes for publication, scheduled review, and definitive expiry. All three fields contribute to temporal validity scoring. |
| `articleType` | `'deep-dive' \| 'primer' \| 'prediction' \| 'analysis' \| 'news'` | Yes | Content classification. Agents use this to select sources appropriate to their task: primers for onboarding, deep-dives for technical detail, predictions for forward-looking synthesis. |
| `predictions` | `Prediction[]` | No | Forward-looking claims made in this article, each with a confidence score, target date, and resolution status. Enables prediction tracking across publishers. |
| `contradicts` | `string[]` | No | conceptIds whose claims conflict with this article's claims. Publishers are expected to maintain this honestly. Agents use contradiction edges to surface disagreements. |
| `supersedes` | `string[]` | No | conceptIds that this article replaces or updates. Enables agents to prefer the most current source in a chain of revisions. |
| `citations` | `Citation[]` | No | Sources supporting the article's claims, each with title, URL, publication date, and author. |

The `conceptId` field deserves special emphasis. It must be kebab-case and stable. Publishers should treat conceptIds as permanent identifiers — like DOIs for academic papers — rather than editable slugs. If the content of an article changes substantially enough to warrant a new conceptId, the old article should remain published with `confidence: 'outdated'` and a `supersedes` pointer on the new article pointing back. This ensures that downstream edges remain valid even as the knowledge graph evolves.

The `confidence` field represents the publisher's assessment at the time of publication. The values form a semantic scale: `current` means the publisher believes the content is accurate and up to date; `outdated` means it was accurate but is no longer fully reliable; `disputed` means there is active disagreement in the field about the claims made; `speculative` means the content makes forward-looking or unverified claims that the publisher intends to resolve over time. Publishers should automate `confidence` updates where possible — for example, a CMS webhook that sets confidence to `outdated` when an article has not been reviewed within its `reviewBy` window.

### 2.2 KnowledgeNode

A KnowledgeNode is a ConceptDNA instance bound to a published URL. It is the atom of the OKP graph: the smallest unit that can be traversed, referenced, and queried. Whereas ConceptDNA is a metadata envelope, a KnowledgeNode is a fully located, resolvable entity in the knowledge graph.

```typescript
interface KnowledgeNode {
  conceptId: string;   // same as conceptDNA.conceptId — repeated for graph traversal convenience
  title: string;       // same as conceptDNA.title — repeated for display without full deserialization
  url: string;         // canonical HTTPS URL of the article
  conceptDNA: ConceptDNA;
}
```

The `url` field must be a canonical HTTPS URL. Redirects are acceptable for backwards compatibility but the canonical URL returned in OKP API responses must be the final destination. The `conceptId` and `title` fields are denormalized from the embedded ConceptDNA for graph traversal performance — agents frequently need to enumerate nodes and render titles without deserializing every ConceptDNA.

When a publisher exposes a KnowledgeNode via any transport profile, the `url` must resolve to an HTML document that contains the same ConceptDNA embedded as a JSON-LD script tag or equivalent. This constraint ensures that the graph representation and the human-readable representation of the content remain synchronized. An OKP validator that fetches a node URL and finds no ConceptDNA, or a ConceptDNA that does not match the API response, must report a compliance failure.

### 2.3 ConceptGraph

A ConceptGraph is a multi-node graph returned in response to traversal queries. It provides a subgraph of the publisher's full knowledge graph, centered on one or more requested nodes, up to a caller-specified depth.

The formal definition is: **G = (V, E, τ)** where:

- **V** is the set of KnowledgeNodes
- **E ⊆ V × V** is the set of directed edges, where each edge (u, v) represents a typed relationship from node u to node v
- **τ: E → EdgeType** is a function that assigns a type to each edge

The EdgeType enumeration defines the legal values for τ:

| EdgeType | Semantics |
|----------|-----------|
| `prerequisite` | The source node is a prerequisite for understanding the target node |
| `enables` | The source node's understanding unlocks the target node |
| `relatedTo` | The source and target nodes are laterally related |
| `contradicts` | The source node's claims conflict with the target node's claims |
| `supersedes` | The source node replaces or updates the target node |

Note that `prerequisite` and `enables` are inverses of each other in the directed sense: if A has `enables: ['B']`, then B has `prerequisites: ['A']`. Implementations must maintain this bidirectional consistency. The OKP validator checks for orphaned edges — edges in E that reference a conceptId not present in V — and reports them as warnings.

The `weight` field on each ConceptEdge (optional, 0–1, defaults to 1.0) allows publishers to express relationship strength. A `relatedTo` edge with weight 0.9 indicates a very close lateral relationship; weight 0.2 indicates a loose association. Weights are informational and not used in compliance scoring, but agents may use them for relevance ranking in graph traversal.

When serializing a ConceptGraph for transport, the `nodes` array must contain complete KnowledgeNode objects (not just conceptIds), and the `edges` array must contain ConceptEdge objects with `from`, `to`, and `type` fields. This self-contained format ensures agents can process the graph without additional API calls.

## 3. Compliance Tiers

OKP compliance is measured on a five-dimensional scoring system, yielding a composite score from 0 to 100. The five dimensions map directly to the five structural deficiencies identified in Section 1. Each dimension is scored independently on a 0–100 scale, and the composite score is the arithmetic mean of all five. Publishers can use the `@okp/validate` package to compute their score programmatically.

**Scoring Dimensions:**

| Dimension | What it measures |
|-----------|-----------------|
| `semantic-identity` | Presence and quality of `conceptId`, `title`, `summary`, and `keyTerms` |
| `relational-context` | Coverage of `prerequisites`, `enables`, `relatedTo` edges |
| `temporal-validity` | Presence of `publishedAt`, `reviewBy`, `expiresAt`, and non-stale `confidence` |
| `confidence-metadata` | Presence of `articleType`, `confidence`, and optional `predictions` |
| `graph-connectivity` | Degree of the node (number of edges), presence of cross-publisher edges |

**Compliance Tiers:**

| Tier | Required composite score | Typical implementation |
|------|--------------------------|------------------------|
| Bronze | ≥ 50 | All required ConceptDNA fields present, at least one edge type populated |
| Silver | ≥ 70 | Required fields present, temporal validity complete, multiple edge types populated |
| Gold | ≥ 90 | All required and most optional fields present, rich graph connectivity, regular `confidence` updates |

Bronze tier represents the minimum viable OKP implementation. An article that provides all required ConceptDNA fields with at least cursory relational context will typically score above 50. Silver tier requires more deliberate authoring: publishers must populate `reviewBy` and `expiresAt` in `temporalValidity`, ensure multiple edge types are represented, and set `confidence` accurately. Gold tier requires the full OKP feature set: predictions where appropriate, citations, `contradicts` and `supersedes` relationships, and high graph connectivity.

The five dimensions are not equally weighted in practice, even though the composite score treats them as equal. A missing `conceptId` is a categorical failure — it prevents graph traversal entirely — whereas a missing `expiresAt` is a minor omission. The validator reports dimension-level scores and per-field failures separately so publishers can prioritize remediation. The composite score is a summary metric for tier classification, not a complete picture of graph quality.

Publishers are encouraged to display their compliance tier on their articles and in their API responses, both as a signal to consumers and as an internal quality incentive. The `@okp/validate` package exposes a `computeCompliance(conceptDNA: ConceptDNA): ComplianceReport` function that returns the score, tier, and a list of failing checks with remediation suggestions. Embedding this in a CMS publishing hook enables automated compliance gates before articles go live.

## 4. Transport Profiles

OKP defines three standard transport profiles for exposing the knowledge graph to AI consumers. These profiles serve different consumer types and different interaction patterns. Publishers are not required to implement all three — Bronze tier requires only that the ConceptDNA is embedded in the HTML; Silver and Gold tiers progressively add transport profile coverage. The profiles are designed to be complementary: a sophisticated AI agent might discover a publisher via llms.txt, fetch graph topology via REST, and then invoke semantic search via MCP.

A critical design principle underlies all transport profiles: **MCP is the transport layer. OKP is the knowledge layer.** MCP defines how tools are invoked and how results are returned; OKP defines what knowledge those tools expose and how it is structured. Publishers implementing OKP over MCP are not required to use any particular MCP server framework — they must only ensure that the five standard tools return OKP-typed responses.

### 4.1 MCP Transport

The MCP transport exposes the OKP graph as a set of Model Context Protocol tools, making it directly callable by any MCP-compatible AI agent or orchestration framework. The endpoint is located at `/api/mcp` on the publisher's domain, following standard MCP server conventions.

Five standard tools are defined for the MCP transport. Publishers may expose additional tools beyond these five, but all Gold-tier implementations must expose all five. The tool signatures use JSON Schema for parameter definition:

| Tool | Signature | Description |
|------|-----------|-------------|
| `search_articles` | `(query: string, limit?: number)` | Full-text and semantic search across the publisher's article corpus. Returns an array of KnowledgeNodes sorted by relevance. |
| `get_article` | `(conceptId: string)` | Retrieve a single KnowledgeNode by conceptId, including full ConceptDNA. |
| `get_related` | `(conceptId: string, limit?: number)` | Retrieve nodes connected by `relatedTo` edges. Returns KnowledgeNodes sorted by edge weight. |
| `get_prerequisites` | `(conceptId: string)` | Retrieve nodes connected by `prerequisite` edges, recursively to any depth. Returns a topologically sorted array. |
| `get_graph` | `(conceptId: string, depth?: number)` | Retrieve a ConceptGraph centered on the given node, up to `depth` hops (default: 2). |

The `search_articles` tool is the primary entry point for agents that do not yet know which conceptId to query. Implementations should support both lexical and semantic (embedding-based) search where possible; if only one mode is available, lexical search is the fallback. The `limit` parameter defaults to 10 and must be capped at 100 to prevent runaway responses.

The reference implementation is the `@okp/mcp-server` package, which provides a ready-to-deploy MCP server that wraps any OKP-compliant data source. Publishers using Sanity, Contentful, or other headless CMSes can configure `@okp/mcp-server` with a thin adapter layer rather than implementing the tool handlers from scratch. The package handles request routing, response serialization, and error formatting according to MCP conventions while ensuring all responses conform to OKP schema types.

Authentication for the MCP transport is out of scope for the v0 specification. Publishers may implement any authentication scheme supported by their MCP framework. The reference implementation supports Bearer token authentication via the standard `Authorization` header.

### 4.2 REST Transport

The REST transport exposes the OKP graph via standard HTTP GET endpoints. It is designed for programmatic consumers that prefer direct HTTP over the MCP tool invocation protocol — integration scripts, CI pipelines, CMS plugins, and lightweight agents that do not require the full MCP session model.

Two endpoints are defined:

**Article endpoint:**
```
GET /api/okp/v0/article?conceptId=<id>
```
Returns a single `Article` object with the full `ConceptDNA` embedded. The response must include the `content` field (plain-text article body for full-text retrieval) in addition to the metadata. Response Content-Type is `application/json`.

**Graph endpoint:**
```
GET /api/okp/v0/graph?conceptId=<id>&depth=<n>
```
Returns a `ConceptGraph` object containing all nodes and edges within `depth` hops of the requested node. The `depth` parameter defaults to 2 and must be capped at 5 to prevent exponential response sizes. For large graphs, publishers should implement cursor-based pagination on the nodes array, with a `nextCursor` field in the response envelope.

Both endpoints must return appropriate HTTP status codes: `200 OK` for successful responses, `404 Not Found` if the conceptId does not exist, `400 Bad Request` for malformed parameters, and `429 Too Many Requests` with a `Retry-After` header for rate-limited responses. Error responses must use a consistent JSON error envelope: `{ "error": { "code": string, "message": string } }`.

CORS headers are required on both endpoints to allow browser-based agents and development tooling to query the API directly. Publishers should at minimum support the `Access-Control-Allow-Origin: *` header for read-only GET endpoints, or a more restrictive origin list if cross-origin access should be limited.

The REST transport is stateless and cacheable. Publishers should set appropriate `Cache-Control` headers. Article responses can typically be cached for minutes to hours depending on publication frequency. Graph responses should have shorter cache TTLs because edge additions from newly published articles immediately affect graph topology.

### 4.3 llms.txt Transport

The llms.txt transport provides a plain-text representation of the publisher's knowledge graph, following the llms.txt specification (originally proposed by Anthropic) for AI crawler consumption. Two files are defined:

**`/llms.txt`** — a structured text file with one-line summaries per article. Each line follows the format:
```
- [Title](URL): Summary sentence.
```
The file must also include a top-level `# Site Name` heading and a brief `> description` paragraph before the article list, following the llms.txt spec. This file is the entry point for AI crawlers that respect the llms.txt convention; it should be updated automatically whenever new articles are published.

**`/llms-full.txt`** — the full-content variant, containing complete article text for each published node. Each article section is separated by a Markdown heading with the article title, and each section includes the full ConceptDNA as a fenced JSON block before the article body. This allows AI systems that ingest the full text to have structured metadata collocated with the prose.

The reference implementation is the `@okp/llms-txt` package, which generates both files from a ConceptGraph data source on demand or on a scheduled rebuild. Publishers using static site generators can output these files at build time; publishers using dynamic CMS systems should generate them via an API route with appropriate caching.

The llms.txt transport is intentionally simple and low-infrastructure: it requires only static file serving, works without authentication, and is compatible with any HTTP client including basic `curl` requests. It is not suitable for dynamic graph traversal — for that, use the MCP or REST transports. Its value is in broad AI crawler compatibility and the extremely low barrier to adoption.

## 5. Versioning Policy

OKP follows semantic versioning with an explicit stability model that distinguishes between the schema (ConceptDNA field definitions), the transport interfaces (API endpoint contracts), and the compliance scoring system (dimension weights and tier thresholds).

The current version is `0.0.1-preview`. The v0 major version designation means the specification carries **no stability guarantees**. Fields may be added, renamed, or removed between v0 minor versions without advance notice, though the authors will maintain a changelog. Publishers implementing v0 should treat it as an early-adopter program and expect to absorb breaking changes. The reference implementations (`@okp/schema`, `@okp/validate`, `@okp/mcp-server`, `@okp/llms-txt`) will release corresponding breaking-change updates as the specification evolves.

v1 is the first stable milestone. The v1 specification will be locked: no required fields will be renamed or removed, no API endpoint paths will change, and no tier threshold will decrease. v1 will be published only after the v0 design has been validated in at least two independent production deployments and the feedback from those deployments has been incorporated. The target date for v1 is 2026 Q4, though this is subject to change.

The rules for schema compatibility across all stable versions (v1 and beyond) are as follows:

- **Adding optional fields** is backwards-compatible. Consumers that do not recognize a new optional field must ignore it without error. This is a minor version increment.
- **Adding required fields** is a breaking change. Existing publishers that do not provide the new required field will fail validation. This requires a major version bump and a migration period with dual support.
- **Removing or renaming any field (required or optional)** is a breaking change. Consumers that depend on the removed field will break. This requires a major version bump with a minimum six-month deprecation window during which both names are accepted.
- **Changing an EdgeType enum value** is a breaking change. Consumers that pattern-match on edge types will produce incorrect behavior with an unrecognized value. This requires a major version bump.
- **Changing compliance tier thresholds** is a minor version increment, but publishers must be given a 90-day notice window before a threshold increase takes effect. A threshold decrease (making it easier to achieve a tier) takes effect immediately.

Version negotiation in the REST transport is handled via URL path (`/api/okp/v0/...`, `/api/okp/v1/...`). Publishers are expected to maintain previous version endpoints for a minimum of 12 months after a new major version is released. The MCP transport includes a `okp_version` field in all tool responses, allowing consumers to detect the schema version without out-of-band negotiation.

## 6. Relationship to Existing Standards

OKP is designed as a complementary layer on top of existing web and metadata standards, not a replacement for them. Publishers who already implement schema.org markup, RSS feeds, llms.txt, or Dublin Core metadata should find that OKP adds structure in the dimensions those standards leave blank. The following table summarizes the relationship:

| Standard | Layer | Problem it solves | What OKP adds |
|----------|-------|-------------------|---------------|
| RSS 2.0 | Syndication | Content discovery and distribution | Knowledge structure: typed relationships, graph traversal, conceptIds |
| JSON-LD / schema.org | Search discovery | Machine-readable metadata for search engines | Relational graph topology, confidence levels, temporal validity beyond `dateModified` |
| llms.txt | AI crawler discovery | Site structure and content inventory for LLM crawlers | Semantic content typing, prerequisite/enables edges, graph queryability |
| MCP | Tool invocation | Standardized protocol for AI agent tool calls | Knowledge payload semantics — what the tools return, not how they are invoked |
| Dublin Core | Metadata | Bibliographic metadata (title, creator, date, subject) | Relational context, epistemic confidence, AI-native fields like `keyTerms` and `articleType` |

The relationship with JSON-LD / schema.org deserves elaboration. Schema.org's `Article` type provides `headline`, `author`, `datePublished`, `keywords`, and `about` — a solid foundation for SEO and search indexing. OKP does not replace these fields; a fully compliant OKP publisher should provide both schema.org markup for search engine compatibility and OKP ConceptDNA for AI agent compatibility. The ConceptDNA can be embedded as a custom type within a JSON-LD script tag using the `@okp/schema` vocabulary namespace, though this is not required — a separate `<script type="application/json" data-okp="conceptDNA">` tag is equally valid.

The relationship with MCP requires careful framing because MCP is both a dependency and a complement. MCP (Model Context Protocol) is a standardized RPC protocol for AI agent tool invocation developed by Anthropic. OKP uses MCP as one of its three transport profiles. But OKP is not an MCP extension — it is a knowledge layer specification that can be transported over MCP, REST, or plain files. The OKP contribution is not the invocation mechanism but the structured knowledge semantics: the ConceptDNA schema, the graph type system, the compliance scoring, and the EdgeType vocabulary. A publisher could implement the five standard OKP tools over any RPC framework and be OKP-compliant; MCP is the preferred transport because it is widely adopted in the AI agent ecosystem.

The relationship with Dublin Core is largely historical. Dublin Core's fifteen elements (`dc:title`, `dc:creator`, `dc:subject`, `dc:description`, `dc:date`, etc.) were designed for library cataloging in the 1990s and remain widely used for bibliographic metadata. OKP's `title`, `summary`, `category`, and `temporalValidity.publishedAt` overlap semantically with Dublin Core fields. Publishers who expose Dublin Core metadata today can treat OKP as a superset: OKP adds the relational and epistemic dimensions that Dublin Core explicitly scoped out of its original design. There is no conflict, and publishers may expose both vocabularies simultaneously.

The `contradicts` and `supersedes` edge types have no direct equivalent in any of the standards listed above. These are novel contributions of OKP, motivated by the AI agent use case: an agent synthesizing knowledge across sources needs to know not just that two articles are related, but that they make conflicting claims or that one has been superseded. The absence of contradiction and supersession metadata in existing standards is a significant gap for agents that aim to reason accurately across large corpora. OKP treats honest declaration of contradictions and supersessions as a first-class obligation of publishers, not an afterthought.
